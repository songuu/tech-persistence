# /review — 多视角审查模式

对当前变更进行多角度审查。灵感来自 Compound Engineering 的 12 子代理并行审查和 gstack 的角色分离。

## 输入
- 无参数：审查当前 `git diff` 中的所有变更
- `$ARGUMENTS` 指定文件或目录：只审查指定范围

## 审查维度

依次从 5 个视角审视代码（每个视角独立思考，不受前一个影响）：

### 视角 1: 安全审查员 🔒
- SQL 注入、XSS、CSRF 风险
- 密钥/token 硬编码
- 输入验证和转义
- 权限检查遗漏
- 依赖漏洞

### 视角 2: 性能工程师 ⚡
- N+1 查询
- 不必要的全量加载
- 缺少索引的查询条件
- 内存泄漏风险
- 可以缓存但未缓存的计算

### 视角 3: 架构审查员 🏗️
- 是否遵循项目现有模式（读取 `.claude/rules/architecture.md`）
- 分层是否正确（Controller/Service/Repository 边界）
- 是否引入了不必要的耦合
- 是否违反 DRY/YAGNI
- 是否与高置信本能一致

### 视角 4: 代码质量 📝
- 命名是否有业务语义
- 函数是否过长（>50 行警告）
- 错误处理是否完整
- 类型是否正确（无 any）
- 边缘情况是否处理

### 视角 5: 测试覆盖 🧪
- 核心路径是否有测试
- 边缘情况是否有测试
- mock 是否合理（不过度 mock）
- 测试是否可维护

## 输出格式

```markdown
## 审查报告

### 🔴 P0 — 必须修复（阻断发布）
| # | 视角 | 文件:行 | 问题 |
|---|------|---------|------|

### 🟡 P1 — 建议修复（本次修复）
| # | 视角 | 文件:行 | 问题 |
|---|------|---------|------|

### 🟢 P2 — 可选优化（记录到 backlog）
| # | 视角 | 文件:行 | 问题 |
|---|------|---------|------|

### ✅ 亮点（做得好的地方）
- ...

### 总评
[1-2 句总结] 置信度: [可以发布/需要修复/需要重做]
```

## 审查后流程
- P0 问题：立即进入修复，不需确认
- P1 问题：列出后等待用户确认是否修复
- P2 问题：记录到 todos 或 backlog
- 全部处理后：建议执行 `/compound` 提取经验

## 与本能系统的集成
- 审查时读取 `.claude/rules/` 中的已知模式和反模式
- 发现新的反模式 → 在报告中标注 `[🧠 新发现]`
- 这些新发现会在 `/compound` 步骤中被提取为本能
