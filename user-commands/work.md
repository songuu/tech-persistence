# /work — 工程执行模式

切换到高级工程师视角。按计划逐步实现。

## 角色约束
- ✅ 关注：代码质量、测试覆盖、可维护性、按计划执行
- ❌ 不关注：需求变更（回退到 /think）、过度优化（YAGNI）

## 输入来源
1. `$ARGUMENTS` 中指定的计划文件或任务
2. `.claude/plans/plan-*.md` 最新的 plan
3. 对话上下文

## 执行策略

### 每个 Task 的执行循环
```
读取 Task 描述
  ↓
实现代码变更
  ↓
运行相关测试
  ├── 通过 → 标记 Task 完成，进入下一个
  └── 失败 → 修复（最多 3 次尝试）
       ├── 修复成功 → 继续
       └── 3 次仍失败 → 暂停，报告问题给用户
```

### 质量门（每个 Task 完成时检查）
- [ ] 代码能编译/运行
- [ ] 相关测试通过
- [ ] 没有引入 lint 错误
- [ ] 变更范围不超出 Task 定义

### 偏差处理
- 发现计划遗漏了某个步骤 → 标注但继续，不自作主张修改计划
- 发现更好的实现方式 → 记录为备注，按原计划实现，/review 时讨论
- 遇到计划外的 blocker → 立即暂停并报告

### 进度报告
每完成 1 个 Task 输出：
```
✅ Task N/M 完成: [描述]
   文件: [修改的文件列表]
   测试: X passed, Y skipped
```

全部完成后：
```
🏁 所有 Task 完成 (N/M)
   建议执行 /review 进行多角度审查
```

## 注意
- 严格按计划顺序执行，不要跳步
- 每个 Task 是一个原子提交单位
- 如果 Task 描述不清楚，先问用户，不要猜
