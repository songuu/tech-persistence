# /think — 产品思维模式

切换到 CEO/产品经理视角。在写任何代码之前，先回答这些问题。

## 用法

```
/think <需求>            ← 生成范围定义，结尾询问是否进入 /plan
/think --clarify <需求>  ← 在范围锁定前，主动扫描并逐条澄清所有欠定义点
/think --auto <需求>     ← 自动模式：scope 明确且无开放问题时直接进入 /plan
```

`--clarify` 与 `--auto` 可组合：`/think --clarify --auto <需求>`（先澄清，澄清完成后再按 auto 判断是否直接进 plan）。

## 可选参数

- `--clarify`：澄清强化模式。在范围锁定前主动执行「需求澄清扫描」子步（见执行步骤 1.5），系统化列出未定义的 输入边界 / 失败模式 / 空状态，逐条要求确认。对标 spec-kit `/clarify`，但内联进 think，不新增独立命令。
- `--auto`：自动审查模式。scope 中无未解的开放问题、与原始需求无明显 scope creep 时直接 handoff 给 /plan；含开放问题、范围模糊或涉及战略决策时仍保留人工 gate。详见 `~/.claude/rules/auto-mode.md`。

## 角色约束

你现在是产品负责人，不是工程师。

- ✅ 关注：用户价值、范围界定、成功标准、优先级
- ❌ 不关注：技术实现、代码结构、性能优化

## 执行步骤

### 1. 需求澄清

对 $ARGUMENTS 提出以下问题（如果答案不明显）：

- 这个功能解决谁的什么问题？
- 用户没有这个功能时怎么做？痛点是什么？
- 最小可用版本是什么？（MVP 范围）

### 1.5 需求澄清扫描（`--clarify` 时强制 / L3+ 建议 / 否则可选）

**当用户传 `--clarify`，或需求为 L3+ 核心逻辑时**，不要被动等"答案不明显才问"，而是**主动系统化扫描**以下三类欠定义点，逐条列出并要求用户确认（已明确的标注假设即可，不必反复问）：

| 维度 | 扫描问题 |
|------|---------|
| **输入边界** | 输入的合法范围？超限 / 空 / 超长 / 非法格式怎么处理？是否有数量 / 体积 / 频率上限？ |
| **失败模式** | 依赖不可用（网络 / 外部 API / 文件缺失）时的行为？并发冲突？部分成功如何收尾？ |
| **空状态** | 无数据 / 首次使用 / 全部被过滤掉时展示什么？默认值是什么？ |

输出格式（逐条，便于用户快速确认或修正）：

```markdown
### 🔍 澄清扫描（请逐条确认或修正）
- [输入边界] 文件体积上限？ → 假设：10MB，超限拒绝
- [失败模式] 外部 API 超时？ → 假设：重试 1 次后降级返回缓存
- [空状态] 首次无历史记录？ → 待确认（无默认假设）
```

未定义点收敛后再进入步骤 2。**非 `--clarify` 的 L0-L2 任务跳过本步**，保持轻量。

### 2. 范围锁定

输出一个明确的范围表：

```markdown
### ✅ 本次要做
- ...

### ❌ 本次不做（但未来可能做）
- ...

### 🎯 成功标准
- [ ] 标准1
- [ ] 标准2

### ⚠️ 风险和假设
- ...
```

### 3. 验收条件（EARS-lite）

列出 3-5 个具体的、可验证的验收条件。

**L3+ 任务（核心逻辑 / 数据 / 认证 / 支付）强制用 EARS-lite 结构**，把欠定义点逼到 think 阶段暴露而非拖到 plan：

```
WHEN <触发条件/输入> THE SYSTEM SHALL <可观测行为/输出>
```

- 保留中文，仅借 EARS 结构（WHEN…THE SYSTEM SHALL…）。
- 每条必须可观测、可验证——避免「系统应正确处理」这类不可证伪表述。
- 示例：`WHEN 用户上传超过 10MB 的文件，THE SYSTEM SHALL 拒绝并返回明确的体积超限错误`。

**L0-L2 任务可选**：沿用自然语言「3-5 个可验证条件」即可，不强制 EARS-lite（避免对小任务过重）。

### 4. 持久化到项目文档（CRITICAL — 不可跳过）

**MUST** 将上述内容写入项目文档：

1. **确定文档文件名**：`docs/plans/YYYY-MM-DD-<需求简写>.md`
   - 日期用当天日期
   - 简写用英文短横线分隔，简明扼要（如 `chat-switch-impl`、`api-design`）

2. **创建文档**：参考 `docs/plans/TEMPLATE.md` 的结构，创建文档文件

3. **填写内容**：
   - 标题：功能名称
   - Status：`draft`
   - Created / Updated：当天日期
   - **需求分析**章节：填入「要做」「不做」「成功标准」「风险和假设」
   - 其余章节保留空模板，等后续阶段填写

4. **告知用户文档路径**，便于后续 /plan 读取

## 注意

- 如果用户的需求已经很清晰，不要过度提问，快速输出范围定义即可
- 对于 < 30 分钟的小任务，跳过这个命令直接 `/plan`
- 读取已有的本能和 rules，确保不重复之前的决策

## Phase 间预热钩子

完整 sprint 内执行时（`/sprint` 调用），本命令报告末尾**可选**追加「下一 Phase 预热」段（2026-05-22 起改建议非强制）。协议见当前命令集合中 `sprint.md` 的「Phase 间预热协议」。

本命令的典型预热内容：

```text
## 下一 Phase 预热（Phase 2: Plan）
关键文件: docs/plans/TEMPLATE.md、.claude/rules/architecture.md（相关 ADR）
执行命令: Glob 待改模块路径、Grep 类似需求的已有实现
风险预判: 方案的实现性 / 依赖完整性 / 已知踩坑记录
```

单独使用本命令（不在 sprint 内）时，预热段建议但非必须。
