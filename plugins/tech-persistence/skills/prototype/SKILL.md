# Prototype

Codex CLI currently registers plugin bundles as skills, apps, and MCP servers. It does not register custom plugin `commands/*.md` files as interactive slash commands in the TUI, so use this skill as the supported Codex entry point for the former `/prototype` command.

## Invocation

Use `$prototype <arguments>` or select this skill through Codex's `@` picker. Treat the user's text after the skill name as the command arguments.

When the command instructions below mention `/prototype`, interpret that as this `$prototype` skill invocation while running in Codex.

## Command Instructions

# /prototype — 原型需求收敛

将原型截图转化为可执行的实现需求。

## 核心原则：假设驱动，不是问题驱动

```
❌ 错误模式 (OpenSpec 式):
   Codex: "点击行后是什么行为？A) 跳转 B) 抽屉 C) 弹窗？"
   Codex: "空状态显示什么？A) 插画 B) 文字 C) 引导？"
   Codex: "分页方式？A) 页码 B) 滚动 C) 加载更多？"
   → 用户回答 15 个抽象问题 → 崩溃

✅ 正确模式 (假设驱动):
   Codex: "我分析了截图，这是我的理解：
     - 点击行 → 跳转详情页 ⚡假设
     - 空状态 → 通用空组件 ⚡假设
     - 分页 → 后端分页，每页 20 条 ⚡假设
     上述有不对的地方吗？"
   → 用户扫一眼: "第2个改成抽屉" → 完成
```

**让用户纠偏比让用户回答简单 10 倍。**

## 子命令
- `/prototype` — 分析截图，输出带假设的完整方案
- `/prototype refine` — 用户纠偏后刷新方案
- `/prototype compare` — 对比原型和实现
- `/prototype checklist` — 生成验收清单
- `/prototype status` — 查看收敛状态

## 执行策略

### 第一步：一次性输出完整方案（带假设标记）

收到截图后，不问任何问题，直接输出一份完整方案。
所有不确定的地方用 `⚡假设` 标记，确定的地方用 `✅确认` 标记。

```markdown
## 原型分析: [页面名称]

### 页面概览
- 类型: 列表管理页 ✅确认
- 模块: 用户管理 ✅确认 (从导航高亮推断)

### 布局
┌─────────────────────────────┐
│ 顶栏: Logo + 导航 + 用户头像  │ ✅确认
├────────┬────────────────────┤
│ 侧栏    │ 搜索栏 + 新建按钮    │ ✅确认
│ 菜单    │ 表格 (5列)          │ ✅确认
│        │ 分页                │ ✅确认
└────────┴────────────────────┘

### 交互方案
| 操作 | 方案 | 置信度 |
|------|------|--------|
| 点击表格行 | 跳转 /users/:id 详情页 | ⚡假设 |
| 点击"新建" | 弹窗表单 (Modal) | ⚡假设 |
| 点击"删除" | Popconfirm 二次确认 | ⚡假设 |
| 搜索 | 后端查询，回车触发 | ⚡假设 |
| 分页 | 后端分页，20条/页 | ⚡假设 |
| 表格排序 | 创建时间可排序 | ⚡假设 |

### 状态方案
| 状态 | 方案 | 置信度 |
|------|------|--------|
| 空状态 | 通用 Empty 组件 + "暂无数据" | ⚡假设 |
| 加载中 | 表格 Skeleton | ⚡假设 |
| 请求失败 | 顶部 message.error 提示 | ⚡假设 |
| 表单验证 | 字段下方红色提示 | ⚡假设 |

### 数据结构 (从表格列推断)
| 字段 | 类型 | 备注 |
|------|------|------|
| name | string | ✅确认 |
| phone | string | ⚡假设: 脱敏显示 138****1234 |
| status | enum | ✅确认: 启用/禁用 |
| createdAt | datetime | ✅确认 |

### ⚡ 假设汇总 (只需看这里)

我做了 8 个假设，请扫一眼，**只回复不对的编号和正确方案**:

1. 点击行 → 跳转详情页
2. 新建 → 弹窗表单
3. 删除 → Popconfirm 确认
4. 搜索 → 后端查询，回车触发
5. 分页 → 后端，20条/页
6. 空状态 → 通用 Empty 组件
7. 加载态 → Skeleton
8. 手机号 → 脱敏显示

如果全对，回复 "ok" 直接进入 /plan。
如果有不对的，比如: "2改抽屉表单 5改前端分页" 即可。
```

### 第二步：用户纠偏（最轻量的交互）

用户可能的回复：
- `"ok"` → 全部假设接受，直接进入 /plan
- `"2改抽屉 6改有插画的空状态"` → 修正 2 个假设
- `"基本对，但还有个权限区别：管理员能看到删除按钮，普通用户看不到"` → 补充遗漏

Codex 收到纠偏后执行 `/prototype refine`，**不重新输出整个方案**，只输出变更摘要：

```markdown
## 方案更新

已修正:
  #2: 新建 → 弹窗表单 → ~~弹窗~~ 右侧抽屉表单 (Drawer)
  #6: 空状态 → ~~通用 Empty~~ 定制插画 + 文字 + 新建按钮

已补充:
  +权限: 删除按钮仅管理员可见

当前状态: 8 个假设中 6 个确认、2 个已修正、1 个补充
→ 可以进入 /plan 了吗？
```

### 第三步：进入 /plan（或继续纠偏）

通常 1-2 轮纠偏即可收敛。如果涉及复杂的多页面流转，可能需要 3 轮。
**但每轮的用户负担是"扫一眼 + 说哪里不对"，而不是"回答 15 个选择题"。**

## 假设置信度来源

假设不是瞎猜。Codex 从以下来源构建假设：

### 1. 项目已有规范 (最高优先)
读取 `.codex/rules/prototype-conventions.md`（如果存在）：
```
组件映射: 表格 → ProTable, 弹窗 → Modal
默认交互: 列表行点击 → 抽屉, 删除 → Popconfirm
默认状态: 空态 → EmptyState 组件, 加载 → Skeleton
```
如果项目规范里定义了，直接标 `✅确认`，不标 `⚡假设`。

### 2. 本能系统 (次高优先)
高置信本能（>=0.7）直接作为默认假设：
- `prefer-drawer-over-modal (0.8)` → 新建表单默认用 Drawer
- `always-backend-pagination (0.9)` → 分页默认后端

### 3. 截图推断 (中等)
从视觉元素推断：
- 看到分页器组件 → 有分页
- 看到搜索框 → 有搜索功能
- 看到操作列有图标 → 有操作按钮

### 4. 行业惯例 (兜底)
无其他依据时用行业通用做法：
- 列表页 → 后端分页 20 条
- 删除 → 二次确认
- 表单 → 字段下方验证

## 多页面场景

上传多张截图时，先输出**页面关系图**（不问问题），再逐页输出方案：

```markdown
## 页面关系 (从截图推断)

截图1 [用户列表] --点击行--> 截图3 [用户详情] ⚡假设
截图1 --点击新建--> 截图2 [新建表单] ⚡假设
截图4 [系统设置] -- 独立入口 ✅确认

⚡ 关系假设正确吗？不对的请说明。

---

## 页面 1/4: 用户列表 (截图1)
[完整方案...]

## 页面 2/4: 新建表单 (截图2)
[完整方案...]
```

对于多页面，Codex 在一条消息中输出**所有页面的方案**，用户一次性纠偏，不需要一页一页来回。
如果截图超过 5 张，拆成两条消息（每条 2-3 页），避免太长。

---

## 复杂度自适应

不需要用户关心"复杂度评估"和"轮次规划"——系统内部自动处理：

| 情况 | Codex 的行为 | 用户感受 |
|------|-------------|---------|
| 1 页简单 CRUD | 一次输出方案 + 假设汇总 | 回复 "ok" 就完事 |
| 3 页多交互 | 输出页面关系 + 各页方案 | 扫一遍，改 2-3 个假设 |
| 6+ 页复杂流程 | 先输出页面关系图确认，再分批输出方案 | 分 2 轮，每轮扫+纠偏 |

**用户永远不需要回答 "这是什么复杂度" 或 "现在第几轮"。**

## 经验回流

用户纠偏的每一条都是宝贵信号：

- 纠偏 "2改抽屉" → 创建/强化本能 `prefer-drawer-over-modal`
- 纠偏 "空状态要有插画" → 更新 `prototype-conventions.md`
- 多次纠偏同一类假设 → `/compound` 时自动提示更新默认值

**下次同一项目分析新页面时，之前纠偏过的假设会变成 ✅确认，不再需要纠偏。**
这就是复利——用得越多，假设越准，纠偏越少，最终趋向于上传截图→"ok"→/plan。
