# 测试策略技能

## 核心原则：测试深度跟着风险走

不是所有代码都需要同等测试。测试过度浪费时间，测试不足引入风险。
正确做法：**评估每次变更的风险等级，匹配对应的测试深度。**

## 风险评估矩阵

对每次代码变更，从 3 个维度评估：

### 维度 1: 影响面
| 等级 | 特征 | 示例 |
|------|------|------|
| 低 | 改动封闭在单一文件/组件内，无外部依赖 | 修改按钮样式、调整文案 |
| 中 | 跨 2-3 个模块，或修改了共享工具函数 | 修改表单验证逻辑、调整列表筛选 |
| 高 | 修改核心模块/底层架构/数据模型/API 接口 | 修改认证流程、支付逻辑、数据库 schema |

### 维度 2: 可逆性
| 等级 | 特征 | 示例 |
|------|------|------|
| 低风险 | 改错了可以立即回滚，无副作用 | UI 文案、样式、静态页面 |
| 中风险 | 回滚需要额外操作但数据不丢 | API 响应格式变更、缓存策略调整 |
| 高风险 | 改错了会导致数据损坏或资金损失 | 数据库迁移、支付扣款、权限变更 |

### 维度 3: 变更类型
| 类型 | 基础风险 |
|------|---------|
| 新增功能 | 中（无回归风险，但有正确性风险）|
| 修改现有逻辑 | 高（有回归风险）|
| 重构（不改行为）| 中（行为不变 = 测试应全过）|
| Bug 修复 | 中（必须有回归测试）|
| 样式/文案 | 低 |
| 配置变更 | 看影响面 |
| 删除代码 | 中（可能有隐式依赖）|

### 综合评分 → 测试等级

```
影响面 + 可逆性 + 变更类型 → 综合风险

  低 + 低 + 样式 = L0 (免测)
  低 + 低 + 新增 = L1 (冒烟)
  中 + 中 + 修改 = L2 (标准)
  高 + 中 + 新增 = L3 (严格)
  高 + 高 + 修改 = L4 (全面)
```

## 五级测试深度

### L0 — 免测
**适用**: 纯样式/文案/注释/格式化
**要求**: 无测试。视觉检查即可。
**产出**: 无测试文件

### L1 — 冒烟测试
**适用**: 简单新增、低风险修改、配置变更
**要求**: 1-3 个测试用例，验证核心路径不崩溃
**产出**:
```
✅ 组件能正常渲染
✅ 主操作能触发（不验证结果细节）
✅ 不抛异常
```
**时间占比**: 变更耗时的 10%

### L2 — 标准测试
**适用**: 常规功能开发、中等风险修改
**要求**: 5-10 个测试用例，覆盖正常路径 + 主要边界
**产出**:
```
✅ 正常路径全覆盖
✅ 空值/空数组处理
✅ 边界值（0、最大值、特殊字符）
✅ 错误处理（无效输入返回预期错误）
```
**时间占比**: 变更耗时的 20-30%

### L3 — 严格测试
**适用**: 核心业务逻辑、API 接口、数据处理
**要求**: 10-20 个测试用例，覆盖正常 + 异常 + 边界 + 并发
**产出**:
```
✅ L2 全部
✅ 异常路径（网络错误、超时、权限不足）
✅ 并发场景（如适用）
✅ 数据一致性验证
✅ Mock 外部依赖
✅ 接口契约测试（请求/响应格式）
```
**时间占比**: 变更耗时的 40-50%

### L4 — 全面测试
**适用**: 支付/认证/权限/数据迁移等高风险变更
**要求**: 20+ 个测试用例，含集成测试
**产出**:
```
✅ L3 全部
✅ 集成测试（多模块协作）
✅ 幂等性验证（重复调用不副作用）
✅ 回滚验证（失败后状态正确）
✅ 性能基准（关键路径响应时间）
✅ 安全测试（SQL注入/XSS/权限绕过）
```
**时间占比**: 变更耗时的 60%+

## 测试框架约定

读取项目 `.codex/rules/testing-patterns.md` 获取：
- 测试框架（Jest/Vitest/Mocha/pytest）
- 测试文件命名约定（`*.test.ts` / `*.spec.ts`）
- Mock 策略偏好
- 测试目录结构

如果项目没有测试框架，在 L2+ 时提示初始化。

## 与本能系统集成

测试相关的本能会影响测试策略：
- `always-test-auth-changes (0.9)` → 认证相关变更自动 L4
- `skip-test-for-i18n (0.7)` → 国际化文案变更自动 L0
- 用户曾纠正测试等级 → 更新风险评估基准
