# 原型需求收敛技能

## 何时触发
- 用户上传了 UI 原型/设计稿图片
- 用户执行了 /prototype
- 用户提到"原型图"、"墨刀"、"设计稿"

## 核心方法论：假设驱动

**不要问问题。输出方案，让用户纠偏。**

纠偏的认知负担远小于回答抽象选择题：
- 回答 "点击行是 A 跳转 B 抽屉 C 弹窗？" → 需要脑中构建场景
- 看到 "点击行 → 跳转详情页 ⚡假设" 然后说 "改抽屉" → 只需判断对不对

## 假设构建优先级

1. 读取 `.codex/rules/prototype-conventions.md`（项目规范）→ 直接 ✅确认
2. 读取高置信本能 (>=0.7) → 直接用作默认值，标 ✅
3. 截图中明确可见的元素 → ✅确认
4. 从截图推断的行为 → ⚡假设
5. 行业通用做法 → ⚡假设

项目规范和本能覆盖得越多，⚡假设越少，用户纠偏越轻松。

## 输出规范

### 必须包含
- 布局结构（ASCII 简图）
- 交互方案表（操作→结果→置信度）
- 状态方案表（状态→方案→置信度）
- 数据结构推断
- **假设汇总**（编号列表，只列 ⚡假设，方便用户扫一眼）

### 假设汇总的格式
```
⚡ 假设汇总 (只需看这里)

我做了 N 个假设，只回复不对的编号+正确方案:
1. 点击行 → 跳转详情页
2. 新建 → 弹窗表单
...
全对就回复 "ok"。
```

### 禁止
- 不输出选择题
- 不在方案输出前提问
- 不每个假设都解释为什么这么假设
- 不重复用户已确认的信息

## 纠偏处理

收到纠偏后：
1. 只输出变更部分，不重复未变的
2. 更新假设状态（⚡→✅）
3. 在本能系统中记录纠偏信号
4. 判断是否可以进入 /plan

## 多页面策略

- 先输出页面关系图
- 然后所有页面方案一次性输出
- 用户一次性纠偏所有页面
- 超过 5 页：分 2 批，每批输出后等纠偏

## 复利机制

每次纠偏都是学习信号：
- 即时：修正当前方案
- 短期：本能系统记录偏好
- 长期：高频纠偏写入 prototype-conventions.md → 下次直接 ✅
- 终态：同一项目新页面 → 上传截图 → "ok" → /plan
