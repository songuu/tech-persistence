## 需求分析

### 用户请求原文
> 研究下 <https://github.com/gsd-build/get-shit-done>，看下当前的架构有没有任何可以借鉴的地方

### §0 项目身份界定 (ADR-011 强制前置)

**TP 是什么**：developer-toolchain self-evolution sibling（自学习方法论系统）。
**4 不可妥协原则**：
1. **多运行时 parity** — Claude Code + Codex 双副本必须功能等价
2. **确定性优先** — 文档协议 + pre-commit-check 双层 enforcement，不靠模型每次记得
3. **轻量优先** — 单 sprint 文档 / Obsidian-friendly 单文件流；新增 artifact 必须有强 ROI 论证
4. **Obsidian 兼容** — 所有产出 markdown + frontmatter，wiki-link 优先

**GSD 是什么**（基于 WebFetch + npm）：meta-prompting + context engineering 系统，主打"solve context rot"。
- 6 命令链：`new-project → discuss-phase → plan-phase → execute-phase → verify-work → ship`
- 5 shared artifacts：`PROJECT.md / REQUIREMENTS.md / ROADMAP.md / STATE.md / CONTEXT.md`
- 15+ runtime 支持（Claude Code / OpenCode / Gemini CLI / Kilo / Codex / Copilot / Cursor / Windsurf / …）
- profile 系统：`core / standard / all`
- `/gsd:surface` 控制 runtime surface
- 主特征：multi-agent orchestration with fresh contexts per executor

**身份差异预判**：
- GSD 是 **meta-prompting framework** 给写产品 app 的工程师用
- TP 是 **自学习方法论** 给 solo-maintainer 维护自己复利系统用
- GSD 解决 "context rot in long product features"
- TP 解决 "knowledge accumulation + cross-runtime + self-evolution"
- 两者**核心解决问题不同**，但**部分技术机制可能正交可借鉴**

### 要做

- 完整探索 GSD 的 6 命令 + 5 artifacts + multi-runtime 机制（不靠 README 一遍读完，至少 Read 3-5 个核心文件）
- 按 ADR-011 4 原则筛选每个机制：可借鉴 / 思想吸收 / 拒绝（含拒绝理由）
- 产出 ≥10 行决策表，每行 reference TP 4 原则之一
- 严格遵循 evidence-based recalibration 协议：每个推荐列**实例计数 + detection signal**
- spawn 3 reviewers（product-lens 强制）做即时闭环
- 若发现 PR-size 脊椎且 reviewer 通过 → 同 sprint 实施
- 最终产出 solution 文档 + 必要 instincts

### 不做

- 不默认建 framework patch（**反模式：sibling-evaluation-defaults-to-framework-building**）
- 不照搬 GSD 多文件 artifact 模式（违反"轻量优先"，已有 spec-kit 评估同结论）
- 不实施 `/gsd-ship` PR 化（违反 "solo-maintainer" 身份假设）
- 不扩展支持 OpenCode/Kilo/Cursor 等额外 runtime（**identity: TP 只承诺 Claude+Codex parity**）
- 不重写 sprint 协议为 phase-based（TP 5-phase 已就位）
- 不修复 GSD 项目本身的 issue（无关本 sprint）

### 成功标准

- [ ] sprint 文档 ≥10 行决策表，每行明示 TP 4 原则
- [ ] 每个 P0/P1 推荐列实例计数（grep `debugging-gotchas.md` / `performance.md` / instincts）+ detection signal
- [ ] 3 reviewer 评级行齐全（product-lens / coherence / dogfood-validity）
- [ ] 若有 PR-size 借鉴 → 实施 + 验证；若无 → solution doc 含明确"不实施"清单
- [ ] solution 文档归档 + sibling-eval 协议复利沉淀（若有新教训）
- [ ] sprint 主文档 status 由 draft → reviewing → completed（无 reviewer pass 不进 completed）
