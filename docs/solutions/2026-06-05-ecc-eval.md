# ECC (Enterprise Claude Code) sibling-eval

> 评估请求 = compare + pros/cons，**非实施承诺**（[[feedback_sibling_eval_default_compare_not_borrow]]）。本文已做二次评审；外部规模数字属于易漂移背景，不作为借鉴裁决的承重依据。

## TL;DR

- **ECC = 与 tech-persistence 最近的直系同类**：同品类（agent harness 自进化系统）、同目标用户（dev-tooling）、同原语（skills/instincts/memory/hooks/跨 harness parity/continuous-learning 置信度评分）。同源（两者 instinct 链都源自社区 "Homunculus"）。
- **规模/验证度悬殊**：ECC 是高热度公开同类（workflow agent 曾记录 gh api 口径 207,355★ / 31,832 fork；2026-06-05 spot-check GitHub README/package 公开口径约 182K+★ / 246 skills / 61 agents）。精确 stars/目录计数随时间漂移，本文只把它当"公开验证度很高"的背景事实；裁决不依赖具体数字。TP solo 自用、~30 skills、双运行时、纯文件。
- **裁决：0 borrow / 2 adapt / 11 backlog / 5 reject**（18 候选）。与历史 sibling-eval 一致（gbrain/gsd/claude-mem 均 0 直接借鉴）。
- **最重要 finding 不是"借什么"，是收敛进化**：TP 独立造出了 ECC ~9 个机制，且 TP 版常更确定性/更 code-anchored（4 不可妥协逼出更好选择）。高热度同类与 solo 项目独立收敛到同一批设计 = TP 架构方向的强外部验证。
- **TP 在一个维度领先 ECC**：ECC 虽 11 harness，其**学习层不跨 harness**——observe hook + Haiku daemon 仅 Claude Code；Codex 只挂通用 MCP memory server，作者自述 "zero parity"。TP 的硬非妥协 #1（copyHookLibs + compatReadDirs 双运行时知识层 parity）是该同类**没做到**的。

## 2. 方法论

dynamic workflow（`ecc-sibling-eval`，26 agents，1.85M tok）：7 个 recon agent 并行深挖 ECC 子系统（gh api + 实际源码，非 README 营销）→ 每个候选立即对抗核验（4 不可妥协 + tp_already_has + target_user_fit + P0 痛点，refute-by-default）。

> 透明性：skills-commands 子系统的 3 个 verify agent 未调 StructuredOutput 掉线，其 recon 数据完整，**3 个候选由主循环 grep 自家代码后补判**（C1 承重事实 `propagate-command-changes.js` 命令列表为 CLI 入参非内部手清单、libs 已 glob 派生；adapt#2 gap `validate-codex-plugin.js:382` 只验 .codex-plugin 对 .claude-plugin 零断言——均已 grep 核实）。

> 外部数字边界：ECC stars / fork / skill / agent 数字是时间敏感背景。后续引用必须带 `verified_at` + 命令或来源；没有嵌入命令证据的旧 gh api 数字只能作为"高热度公开同类"旁证，不能承重。

TP P0 痛点（推荐排序依据，[[feedback_research_recommendation_sort_by_user_pain]]）：
- **P0a** documented-claim-vs-code-reality drift（TP #1 回归源，[[documented-claim-vs-code-reality-drift]]）
- **P0b** 注入知识是否真改变行为（语义遵守不可测，[[ADR-022]]）
- **P0c** 跨运行时 parity 维护负担（每个 lib 需 plugin copy + sha sync + validator）
- **P0d** instinct domain 多样性过低（detectPatterns 只产 workflow/debugging/code-style）
- **P0e** enforcement dead-on-arrival（FP 高→学会 `--no-verify`→enforcement 死亡，[[feedback_enforcement_dead_on_arrival_82pct]]）

## 5. 推荐（按 P0 痛点排序，evaluation 非承诺）

> 用户 persona 接受"什么都不做"为合理结果（[[user_workflow_preferences]]）。以下 2 个 adapt 是**全 18 候选里仅有的、过 4 不可妥协且命中 P0 痛点**的；其余 16 为 backlog/reject。即便这 2 个也低紧迫（measure-before-enforce）。

### 优先级 0（零代码，先做）：测量 P0d / 跨项目复现，验证再投入
在 `~/.claude/homunculus/**/instincts/` 与 `~/.codex/homunculus/**/instincts/` 上统计：(a) instinct 文件名前缀 / domain 分布（量化 P0d 严重度）；(b) 当前项目 id（`detectProjectIdentity()`）对应 instinct 是否存在；(c) 同 instinct id 跨 ≥2 distinct project hash 的数量（决定候选 #3 是否永不触发）。纯读文件、零表面。守 measure-before-enforce（[[ADR-022]]/[[ADR-013]]）——**没有这个数据，#2 和 #3 都是盲投**。

### 推荐 #1（P0a，TP #1 回归源）：.claude-plugin manifest validator 断言 + MCP fq-name 守卫
- **what**：`validate-codex-plugin.js` 加对 `.claude-plugin/plugin.json` 的确定性断言——禁 `agents`/`hooks` key（Claude 自动发现，显式声明会被 validator 拒）、要求 commands/skills 为数组、MCP fully-qualified 工具名 <64 字符；配一份 `PLUGIN_SCHEMA_NOTES.md` 记录这些未文档化约束 + flip-flop 历史。
- **why adapt 非 borrow**：不能照抄 ECC test——TP 两个 manifest **要求集相反**（Codex manifest *要求* hooks/mcpServers，Claude manifest *禁止* agents/hooks），须运行时分别实现（沿 TP 既有 runtime-projection 模式）。
- **gap 已 grep 核实**：validate-codex-plugin.js:382 仅验 .codex-plugin，对 .claude-plugin **零断言**。
- **紧迫度**：低。当前 Claude manifest 已最小合规，最长 MCP fq-name ~55<64——blast radius 潜在非活跃。属"防未来静默 install 失败"的廉价确定性护栏。

### 推荐 #2（P0d，instinct 多样性）：lean `/instinct-mine` repo-convention 采集通道
- **what**：一个 model-driven 命令（类 `/learn`），读 git log + 配置文件，把 repo 约定（commit 前缀、命名风格、测试流、"改 hook 必同步 config+test+docs"）产出为 `source: repo-curation` 的 instinct，填充 detectPatterns 够不到的 git/testing/code-style domain。
- **why adapt 非 borrow**：丢弃 ECC 全部机器（Python daemon + Haiku + 1826 行 instinct-cli.py + auto-promote）；置信度须**折扣进场**（类 instinct-import ×0.8）+ 要 session 确认才升，否则灌爆 12KB 注入预算 + 污染 /evolve 聚类。
- **紧迫度**：低，且**先做优先级 0 测量**。下游 `docs/plans/2026-06-05-ecc-adapts.md` 已补测：P0d 真实但比初判更窄，#2 应收紧为 spike-first；若 mined 输出多为 rules/CLAUDE.md 已手述条目，则本项自我否决。

## 6. 收敛进化 finding（本评估最高价值产出）

TP 在**无任何借鉴**的前提下，独立造出了 ECC 的核心机制，且常更确定性/更 code-anchored：

| 机制 | ECC | TP（独立实现） | TP 是否更优 |
|---|---|---|---|
| instinct 链 observe→evolve | continuous-learning-v2 | 同（同源 Homunculus） | 平 + TP 有 demand-side telemetry |
| 项目身份 git-remote-hash | sha256(remote)[:12] | detectProjectIdentity | 平 |
| 路径分级 FP 抑制 | RuntimeConfidence 7 级 | knowledge-drift classifyReference | TP 更确定性（二元 gate vs 0.25× 模糊分） |
| 秘钥正则扫描 | governance-capture | secret-scan-on-demand + isLikelyExample | 平 |
| context-budget 仪表 | 静态 words×1.3 估计 | renderContextCostSummary | **TP 更准**（真实 char 分配器读数） |
| hook gating in-code | isHookEnabled 纯函数 | hook-registry.js（[[ADR-014]]） | 平 |
| regenerate-and-compare drift | harness-adapter-compliance | checkPropagateSync / knowledge-drift | **TP 更优**（doc-vs-code/byte，ECC doc-vs-doc） |
| per-session 摘要 | raw transcript .tmp | evaluate-session 从 observations | **TP 更高信号**（结构化非 raw） |
| fail-open hook + 标记 | `[Hook] failed:` | hook-exit-codes.md `[name] failed:` | 平 |

**结论**：一个高热度同类与一个 solo 项目独立收敛到同一批设计——这是 TP 架构方向（4 不可妥协）正确性的强外部验证。差异处 ECC 全是 maximalism（大规模 skills / 11 harness / SQLite / GUI / daemon / 102 规则），正是 TP lightweight-first 该拒的。

**TP 领先点**：ECC 虽 11 harness，**学习层不跨 harness**（observe + Haiku daemon 仅 Claude；Codex 挂通用 MCP server，作者自述 "zero parity"）。TP 硬非妥协 #1（copyHookLibs + compatReadDirs 双运行时知识层 parity）是 ECC **没做到**的——TP 用"少 harness 换深 parity"，ECC 用"多 harness 换浅 parity"，是定位差异不是优劣，但说明 TP 的 parity 严格度在该品类罕见。
