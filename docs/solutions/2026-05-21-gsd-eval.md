# gsd-build/get-shit-done 评估 sprint — 0 直接实施 + 5 follow-up + 11 硬拒绝 + 2 防御性拒绝改正

> **Status**: `completed`（Phase 4 三 reviewer 全 DONE_WITH_CONCERNS, 3 P0 + 5 P1 + 2 防御性拒绝全修复; 详见 §Phase 4 reviewer 反馈段）

## Problem

用户请求："研究下 <https://github.com/gsd-build/get-shit-done>，看下当前的架构有没有任何可以借鉴的地方"。

GSD (Get Shit Done) 是 Node.js CLI + SDK 形态的 meta-prompting + context engineering 系统（>=22 Node, npm 全局安装, Vitest + node:test, 15+ runtime 支持），主打"solve context rot"。Marketing 简化为 6 命令链（new-project → discuss-phase → plan-phase → execute-phase → verify-work → ship），**实际命令面是 67 个**（含 audit-fix, audit-uat, eval-review, forensics, ns-* namespace 命令, surface 等独特概念）。

tech-persistence (TP) 是 **自学习方法论 sibling**（[[ADR-011]] 4 不可妥协原则: multi-runtime parity / 确定性优先 / 轻量优先 / Obsidian 兼容），21 命令 + 4 hook + 双 runtime（Claude Code + Codex）副本 + Memory v5 + agentmemory + 5-phase sprint。

## Root Cause（评估外部 sibling project 的 3 个失效模式）

本次评估应用并实证 [[ADR-011]] identity-first + [[evidence-based-recalibration]] 协议:

1. **身份模糊**: 不先回答"本项目是什么" → ROI 用错坐标
   - 缓解: §0 项目身份界定 (TP=self-evolution for solo-maintainer / GSD=meta-prompting for product apps) → 决策表 15 行每行 reference 4 原则之一
2. **脊椎/表面混淆**: 把 GSD 67 命令面 / .planning/* 多文件 artifact / runtime surface 控制等**外在产物**当借鉴脊椎
   - 缓解: TL;DR §1 已识别核心矛盾——marketing 6 命令是简化版, 实际 67 命令本身就是 TP 拒绝的反模式 [[command-bloat]]
3. **拒绝面过宽**: 拒绝整套 GSD 时可能把"plan-checker iteration"这个**正交脊椎**也一并拒绝
   - 缓解: T4 判定 P1 follow-up 而非直接拒绝；Phase 4 reviewer 必挑战此判定

## Solution — 3 层方法论防御 + 5 follow-up + 0 直接实施

### A) Identity-question-first ([[ADR-011]] 重申)

[[ADR-011]] 4 不可妥协原则下的过滤逻辑:
- **多运行时 parity (MR)**: TP 只承诺 Claude+Codex 双 runtime parity. GSD 15+ runtime 的 surface gating 不适合 (#11/12)
- **确定性优先 (DET)**: 优先 pre-commit / hook 拒绝 > 文档协议. GSD plan-checker 概念若实施必须 enforcement 层 (#1)
- **轻量优先 (LT)**: 单 sprint 文档 + Obsidian-friendly 单文件流. GSD .planning/* 多文件 / 67 命令面违反 LT (#10/11/15)
- **Obsidian 兼容 (OBS)**: 单文件 + frontmatter + wiki-link. GSD `~/.claude/get-shit-done/templates/*.md` 多模板复用机制不在 TP 范围

## Sprint Lessons (Phase 4 reviewer 闭环后最终版, 6 教训)

1. **Marketing 简化版 vs 实际命令面差异 = 评估陷阱**: GSD README 6 命令简化版误导本 sprint Phase 1 假设错"6 命令链", 实际 67 命令。**[[ADR-012]] 关键假设验证段必须覆盖此类外在产物 ↔ 实际产物差异**, 不只看 README 一遍。→ Phase 2 H1 修正"6 命令链"为"67 命令", 沉淀到 [[command-bloat]] Evidence。
2. **反向参照是有效借鉴产物类型**: GSD 67 命令面在 TP 是反向参照（TP 已有 [[command-bloat]] 防止此漂移）。**反向参照应明示在决策表（#15 标"TP 已防"）**, 是身份原则被实证的具体反例, 不是"未借鉴"。
3. **跨 follow-up signal merge 是 recalibration 的副产物**: 本 sprint #2 ADR ingest 标"思想关联"等价为 spec-kit follow-up `checkPlanAdrReference` 立项条件 signal +1。**未来 sibling-eval 默认做"跨 follow-up signal merge"**, 不重复立项。
4. **本能引用悬空 = dogfood failure 类型（新教训）**: 本 sprint Phase 3 引用 6 wiki-link 本能, Phase 4 dogfood reviewer grep 发现 0/6 真实注册。**修复路径 = 本 sprint 注册 6 个本能不延期**。沉淀: 任何引用 `[[name]]` 形式本能的 sibling-eval / solution doc, Phase 4 dogfood 必须 grep 验证存在性。
5. **实例计数底数必须实证（新教训）**: Phase 3 凭印象写 "spec-kit 2 次 + mattpocock 1 次"; Phase 4 dogfood grep 实证仅 sprint-speed-layer1 §Sprint Lessons #1 明确 "两次 plan-error" + 本 sprint 1 次 = N=3。**立项条件必须 grep-able 不依赖人主观判断**。所有 5 follow-up 立项条件改 grep-able pattern。
6. **防御性拒绝识别 recalibration §Step2.5 第 2 次实证**: Phase 4 product-lens 发现 #8 secret-scan ("0 实例" 把 enforcement 难度混同需求判断) + #9 verify-work ("solo-maintainer = UAT dogfood" 把执行功能路径混同 systematic gap detection)。**reject 自挑战与 absorb 自挑战同等重要**。第 1 次实证 mattpocock `handoff` 同案。
